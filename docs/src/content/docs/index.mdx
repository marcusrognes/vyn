## What is Vyn?

Vyn is a full-stack TypeScript framework with the smallest reasonable surface
area. File-based routing, typed RPC over HTTP and WebSocket, server-rendered
data loaders, and an auth primitive — all designed to be readable end-to-end.

Every layer is meant to fit in your head. The whole framework is a few
thousand lines. You can pull up any file in the source and understand what
it does without a wiki.

Vyn runs on Deno and Node from a shared codebase. Pick one or use both —
the public API is identical; only the imports and project file differ.

## 30-second tour

Declare an action. Its location in the file tree becomes its RPC path,
its agent tool id, its CLI subcommand. No barrels to maintain, no name
to keep in sync with a path.

```ts
// features/notes/notes.actions.ts
import { createQuery, createMutation, createSubscription, v } from "@vyn/core";
import { NoteSchema } from "../models/note.ts";

export const list = createQuery({
	description: "List notes for the current session.",
	input:  v.object({ limit: v.number().min(1).max(100).default(50) }),
	output: v.array(NoteSchema),
	run: async opts => opts.ctx.db.notes.limit(opts.input.limit).all(),
});

export const onCreated = createSubscription({
	description: "Stream notes as they are created.",
	input:  v.object({}),
	output: NoteSchema,
	run: async function* (opts) {
		for await (const note of opts.events) yield note;
	},
});

export const create = createMutation({
	description: "Create a note.",
	input:  NoteSchema,
	output: NoteSchema,
	run: async opts => {
		const note = await opts.ctx.db.notes.insert(NoteSchema.create(opts.input)).get();
		onCreated.emit(note);   // typed reference; subscribers receive the new note
		return note;
	},
});
```

```ts
// server.ts
import { serve } from "@vyn/server";
import "./_vyn.gen.ts";   // codegen barrel — discovers every *.actions.ts

serve({ port: 8000 });
```

Same code on Deno and Node. Only the project file and install command
differ; the code you write does not. The runtime switcher changes those
parts of the docs but leaves your application code alone.

The client gets a typed proxy whose shape matches the file tree exactly.
No code generation step on the client.

```ts
// public/routes/index.ts
const notes = await rpc.notes.list.query({});
//    ^? Note[]

const fresh = await rpc.notes.create.mutate({ body: "hello" });
//    ^? Note
```

The split between query and mutation is honest: queries are cacheable
and side-effect-free; mutations declare what they affect by importing
the actions they touch and calling their methods. No topic strings, no
metadata drift.

## Design choices

<CardGrid>
	<Card title="No magic" icon="seti:typescript">
		Every behavior is traceable to code you can read. No hidden
		dependency injection, no compile-time macros, no decorators that
		rewrite your functions. The framework is glue between native
		platform primitives — custom elements, signals, fetch, WebSocket.
	</Card>
	<Card title="Read in one sitting" icon="open-book">
		The whole framework is small enough to skim end-to-end before
		dinner. If you can't explain why a request behaves the way it does
		by reading two files, that's a bug in Vyn, not in you.
	</Card>
	<Card title="Deno and Node, one codebase" icon="random">
		Apps target either runtime without rewriting. Runtime adapters live
		at the boundary; user code touches only the shared API.
	</Card>
</CardGrid>
